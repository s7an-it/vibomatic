## Personas

| ID | Name | Archetype | Tier | Key deployment mode | Primary features |
|----|------|-----------|------|-------------------|-----------------|

| P2 | The Team Lead | Engineering manager who needs a shared private tool to guide team learning | 1-Platform-Native | Self-hosted private | Team dashboard, aggregate skill maps, trend-to-planning bridge, multi-user admin |

| P4 | The Career Transitioner | Software developer pivoting into AI/ML who needs industry-aligned learning progression | 2-Near-Future | Hosted SaaS (operator instance) | Skill-level filtering, prerequisite-aware learning paths, progress tracking, beginner-appropriate trend curation |

## Vision Feedback

| Finding | Which persona | Impact | Recommendation |
|---------|--------------|--------|----------------|
| Vision mentions "career transitioners" as secondary users but provides no detail on how the product adapts for different skill levels. | P4 (Career Transitioner) | High. Without skill-level-aware trend presentation and prerequisite-aware learning paths, career transitioners will bounce immediately. The product will feel like it is built for experts only. | Add explicit skill-level filtering and prerequisite logic to the V1 spec. Learning paths must be structured progressions, not flat resource lists. Trend views should annotate difficulty or relevance by user level. |
| Vision describes three deployment modes but does not specify what "hosted-for-profit" actually includes for operators. Multi-tenancy, billing hooks, user management, and white-labeling are not mentioned. | P3 (Operator) | Critical. Without clear operator infrastructure, the hosted-for-profit mode is an empty promise. No serious operator will build a business on a single-user codebase that says "you can host it for others." | Define operator-mode requirements explicitly: multi-tenant data isolation, user management API, billing webhook integration, branding/white-label configuration, and operator analytics dashboard. These must be in the V1 architecture, not deferred. |
| Vision states "personalized skill tracking" but does not describe how skill profiles are created or maintained. No mention of prerequisite mapping between topics. | P1, P4 | Medium-High. P1 expects recommendations to improve after profile configuration. P4 depends entirely on the system knowing what she does not know. Without a clear skill model, personalization is a claim without substance. | Define the skill taxonomy for AI/ML (categories, levels, relationships). Specify how profiles are created (self-assessment, inferred from learning activity, or both). Map prerequisites between topics so learning paths can be sequenced. |
| Vision mentions "team leads and engineering managers" but does not describe any team-specific features beyond "a shared tool." | P2 (Team Lead) | Medium. P2's entire value proposition depends on team features that are currently unspecified: team dashboards, aggregate skill views, shared learning paths, role-based access. | Spec out team features as a distinct feature area: team creation, member management, aggregate skill visualization, shared trend feeds, and admin controls. Clarify whether team features are available in all deployment modes or only self-hosted private. |
| Vision lists opt-in telemetry but does not address how operators get usage analytics for their hosted instances. Operators need analytics to run a business but should not depend on opt-in telemetry from end users. | P3 (Operator) | Medium. Operators need engagement metrics (DAU, trend views, learning path starts, churn indicators) that are distinct from product telemetry. These are business-critical, not optional. | Distinguish between product telemetry (opt-in, for TrendLearner project improvement) and operator analytics (always-on for operator instances, scoped to that operator's users). Operator analytics are a feature, not telemetry. |
| Vision's anti-metric "time spent in app" is philosophically correct but creates a measurement problem. If the product succeeds, users leave quickly. How do you measure value delivery for users who interact briefly? | P1, P4 | Low-Medium. Not a persona-specific issue but affects how success is measured for all personas. Short sessions that lead to external learning are the goal, but they are hard to track. | Define a "learning initiation" event (user clicks through to an external resource) as the primary value signal. Consider optional "did you complete this?" check-ins on return visits. Do not track users on external sites. |
| Vision does not mention mobile access. P4 (career transitioner) and P1 (independent learner) both have use cases for checking trends on mobile devices. | P1, P4 | Low for V1. Mobile is a convenience, not a blocker. But responsive design should be considered from the start to avoid a costly retrofit. | Ensure the web UI is responsive from V1. Do not build a native mobile app, but do not build a desktop-only layout. Forward-looking area #8 (mobile companion app) should be informed by actual mobile usage data from the responsive web app. |
| The vision's "Enterprise L&D teams" secondary user segment is not developed into a persona because it overlaps significantly with P2 (team lead) and would require enterprise features (SSO, SCIM, audit logs) that are not in V1 scope. | N/A | Low for V1. Enterprise L&D is a valid future persona but depends on features not yet specified. P2 covers the team use case at a scale appropriate for V1. | Defer Enterprise L&D persona to a future build-personas run after V1 ships and enterprise features are scoped. P2 serves as a bridge persona that can evolve toward enterprise needs. |
